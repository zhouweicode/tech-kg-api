# 知识图谱表详细说明文档

## 1. 文档目的

本文面向科技知识图谱项目的设计、开发、测试和汇报场景，对图数据库中的“表”进行详细说明。  
在图数据库语境下，“表”通常对应两类对象：

1. 节点表：对应 Neo4j 的 `Label`。
2. 关系表：对应 Neo4j 的 `Relationship Type`。

因此，本文中的“知识图谱表”包含节点表和关系表两部分。 



## 2. 总体设计思路

### 2.1 为什么图数据库里还要讲“表”

关系型数据库一般用二维表描述对象和对象之间的关联；图数据库则把对象表示为节点，把对象之间的关联表示为关系。  
为了便于业务、开发和实施团队统一理解，仍然可以把图数据库中的节点类型和关系类型理解为“表结构”。

例如：

1. `Scholar` 节点表：表示专家/人才实体。
2. `Paper` 节点表：表示论文实体。
3. `DIRECT_REL` 关系表：表示专家与专家之间的直接合作关系。

### 2.2 本项目的建模原则

本项目采用“三层结构”建模：

1. 实体层：描述“人、机构、论文、专利、企业、技术、事件”等稳定对象。
2. 事实层：描述“作者参与、专利发明、项目参与、任职经历、教育经历、企业合作”等带上下文的事实。
3. 汇总层：描述“直接关系、论文合作关系、同事关系、校友关系、间接关系、专家企业关系”等经过加工计算得到的结果。

这样设计的好处是：

1. 原始事实可以追溯。
2. 推理结果可以快速查询。
3. 时间、角色、证据、规则版本等信息不容易丢失。

## 3. 表分类总览

本知识图谱表分为 4 大类：

1. 基础实体节点表
2. 事实节点表
3. 事实关系表
4. 汇总/推理关系表

### 3.1 基础实体节点表

基础实体节点表用于承载核心业务对象，是查询入口和关系汇聚中心。

包含：

1. `Scholar`（专家）
2. `Organization`（机构）
3. `Department`（部门）
4. `Paper`（论文）
5. `Venue`（期刊会议）
6. `Patent`（专利）
7. `Project`（项目）
8. `Enterprise`（企业）
9. `School`（学校）
10. `College`（院系）
11. `Technology`（技术）
12. `IndustrySegment`（产业链环节）
13. `Team`（团队）
14. `IndustryEvent`（产业事件）
15. `CapitalEvent`（资本事件）
16. `Award`（奖项）
17. `UpdateBatch`（更新批次）

### 3.2 事实节点表

事实节点表用于承载复杂上下文事实，尤其是时间、角色、顺序、职责、证据等信息。

包含：

1. `Authorship`（作者事实）
2. `Inventorship`（发明事实）
3. `ProjectParticipation`（项目参与事实）
4. `Employment`（任职经历）
5. `Education`（教育经历）
6. `EnterpriseCooperation`（企业合作事实）
7. `InferencePath`（推理路径）
8. `TrendReport`（趋势报告）

### 3.3 事实关系表

事实关系表把实体节点与事实节点连接起来，或把实体之间的原始事实连接起来。

包含：

1. `HAS_AUTHORSHIP`（拥有作者事实）
2. `AUTHOR_OF`（作者对应论文）
3. `PUBLISHED_IN`（发表于载体）
4. `HAS_INVENTORSHIP`（拥有发明事实）
5. `INVENTOR_OF`（发明对应专利）
6. `HAS_PARTICIPATION`（拥有项目参与事实）
7. `PARTICIPATES_IN`（参与项目）
8. `HAS_EMPLOYMENT`（拥有任职事实）
9. `EMPLOYED_BY`（任职于机构）
10. `IN_DEPARTMENT`（任职于部门）
11. `HAS_EDUCATION`（拥有教育事实）
12. `STUDIED_AT`（就读于学校）
13. `IN_COLLEGE`（就读于院系）
14. `HAS_ENTERPRISE_COOP`（拥有企业合作事实）
15. `COOP_WITH_ENTERPRISE`（与企业合作）
16. `RELATED_TECH`（关联技术）
17. `OWNS_TECH`（掌握技术）
18. `FOCUSES_ON`（研究方向聚焦）
19. `BELONGS_TO_SEGMENT`（属于产业链环节）
20. `PART_OF_SEGMENT`（技术属于环节）
21. `EVENT_IN_SEGMENT`（事件发生于环节）
22. `EVENT_INVOLVES_TECH`（事件关联技术）
23. `INVESTS_IN`（投资于企业）
24. `MEMBER_OF_TEAM`（加入团队）
25. `TEAM_IN_ORG`（团队归属于机构）
26. `WON_AWARD`（成果获得奖项）
27. `LOADED_IN_BATCH`（载入更新批次）
28. `PATH_FROM`（路径起点）
29. `PATH_TO`（路径终点）
30. `HAS_TREND_REPORT`（关联趋势报告）

### 3.4 汇总/推理关系表

汇总/推理关系表用于存放可直接用于分析、展示、统计、排序的结果型关系。

包含：

1. `DIRECT_REL`（直接关系）
2. `PAPER_COOP_REL`（论文合作关系）
3. `COLLEAGUE_REL`（同事关系）
4. `ALUMNI_REL`（校友关系）
5. `EXPERT_ENTERPRISE_REL`（专家企业关系）
6. `INDIRECT_REL`（间接关系）
7. `PAIR_COOP_SUMMARY`（两点合作成果汇总）
8. `EVENT_EXPERT_REL`（事件关联专家）
9. `EVENT_TEAM_REL`（事件关联团队）
10. `UPSTREAM_OF`（上下游关系）
11. `VALUE_FLOW_TO`（价值流向）

## 4. 基础实体节点表详细说明

### 4.1 Scholar 表

用途：表示科技专家、人才，是整个图谱中的核心人物主表。  
典型来源：学者库、机构名录、论文作者消歧结果。  
典型作用：作为直接关系、间接关系、同事关系、校友关系、合作成果、企业关系、产业链关系分析的统一入口。

核心字段说明：

| 字段名 | 中文名称 | 含义 |
| --- | --- | --- |
| `scholar_id` | 专家唯一标识 | 专家主键字段 |
| `name` | 姓名 | 专家姓名 |
| `alias_text` | 别名信息 | 别名、英文名、曾用名等 |
| `gender` | 性别 | 专家性别 |
| `title` | 职称 | 如教授、副教授、研究员 |
| `research_fields_text` | 研究方向列表 | 规范化研究方向集合 |
| `discipline_text` | 学科领域列表 | 所属学科领域集合 |
| `h_index` | H指数 | 学术影响力指标 |
| `citation_count` | 总被引量 | 成果累计被引用次数 |
| `org_name_std` | 标准机构名称 | 规范化后的机构名称 |
| `updated_at` | 更新时间 | 数据最近更新时间 |

典型关系：

1. `Scholar` -> `Authorship`
2. `Scholar` -> `Inventorship`
3. `Scholar` -> `ProjectParticipation`
4. `Scholar` -> `Employment`
5. `Scholar` -> `Education`
6. `Scholar` -> `EnterpriseCooperation`
7. `Scholar` -> `Scholar` 的 `DIRECT_REL`、`PAPER_COOP_REL`、`COLLEAGUE_REL`、`ALUMNI_REL`、`INDIRECT_REL`

对应功能模块：

1. 科技专家/人才直接关系
2. 科技单节点间接关系
3. 科技两点合作成果
4. 科技专家同事关系
5. 科技专家校友关系
6. 科技专家论文合作关系
7. 重点关注科技企业关系
8. 科技产业链全景图

## 5. 事实节点表详细说明

### 5.1 Authorship 表

用途：表示“某位专家作为作者参与某篇论文”的事实。  
为什么要单独建表：如果直接把专家和论文相连，作者顺序、通讯作者、原始署名单位等信息会难以管理。

## 6. 事实关系表详细说明

### 6.1 作者与论文链路

典型链路：

`Scholar` -> `HAS_AUTHORSHIP` -> `Authorship` -> `AUTHOR_OF` -> `Paper`

说明：

1. `HAS_AUTHORSHIP` 表示专家拥有某个作者事实。
2. `AUTHOR_OF` 表示该作者事实对应某篇论文。
3. 这种设计允许保留作者顺序、通讯作者等上下文。

### 6.2 发明人与专利链路

典型链路：

`Scholar` -> `HAS_INVENTORSHIP` -> `Inventorship` -> `INVENTOR_OF` -> `Patent`

说明：用于识别专利合作、企业技术布局和专利证据链。

### 6.3 专家与项目链路

典型链路：

`Scholar` -> `HAS_PARTICIPATION` -> `ProjectParticipation` -> `PARTICIPATES_IN` -> `Project`

说明：用于项目共同承担关系识别与角色分析。

### 6.4 任职链路

典型链路：

`Scholar` -> `HAS_EMPLOYMENT` -> `Employment` -> `EMPLOYED_BY` -> `Organization`

补充链路：

`Employment` -> `IN_DEPARTMENT` -> `Department`

说明：用于同事关系和职业轨迹分析。

### 6.5 教育链路

典型链路：

`Scholar` -> `HAS_EDUCATION` -> `Education` -> `STUDIED_AT` -> `School`

补充链路：

`Education` -> `IN_COLLEGE` -> `College`

说明：用于校友、院友关系推理。

### 6.6 企业合作链路

典型链路：

`Scholar` -> `HAS_ENTERPRISE_COOP` -> `EnterpriseCooperation` -> `COOP_WITH_ENTERPRISE` -> `Enterprise`

补充链路：

`EnterpriseCooperation` -> `RELATED_TECH` -> `Technology`

说明：用于专家企业关系分析和技术合作分析。

### 6.7 产业链与事件链路

典型链路：

1. `Enterprise` -> `BELONGS_TO_SEGMENT` -> `IndustrySegment`
2. `Technology` -> `PART_OF_SEGMENT` -> `IndustrySegment`
3. `IndustryEvent` -> `EVENT_IN_SEGMENT` -> `IndustrySegment`
4. `IndustryEvent` -> `EVENT_INVOLVES_TECH` -> `Technology`
5. `CapitalEvent` -> `INVESTS_IN` -> `Enterprise`

### 6.8 其他辅助关系

1. `MEMBER_OF_TEAM`：专家加入团队
2. `TEAM_IN_ORG`：团队归属于机构
3. `WON_AWARD`：成果获得奖项
4. `LOADED_IN_BATCH`：节点或结果属于某次导入批次
5. `PATH_FROM`、`PATH_TO`：推理路径连接起点和终点专家
6. `HAS_TREND_REPORT`：事件关联趋势报告

1. 节点属性索引：提升按姓名、标题、时间、数值范围检索的效率。

## 10. 典型查询示例

### 10.1 查询所有专家

```cypher
MATCH (n:Scholar)
RETURN n.scholar_id, n.name, n.h_index
LIMIT 20;
```

### 10.2 查询专家直接关系

```cypher
MATCH (a:Scholar)-[r:DIRECT_REL]->(b:Scholar)
RETURN a.name, b.name, r.coop_count, r.confidence
LIMIT 20;
```

### 10.3 查询同事关系

```cypher
MATCH (a:Scholar)-[r:COLLEAGUE_REL]->(b:Scholar)
RETURN a.name, b.name, r.overlap_start, r.overlap_end
LIMIT 20;
```

### 10.4 查询专家与企业关系

```cypher
MATCH (a:Scholar)-[r:EXPERT_ENTERPRISE_REL]->(e:Enterprise)
RETURN a.name, e.enterprise_name, r.relation_type, r.role, r.value_score
LIMIT 20;
```

### 10.5 查询产业链事件与专家

```cypher
MATCH (ev:IndustryEvent)-[r:EVENT_EXPERT_REL]->(s:Scholar)
RETURN ev.event_name, s.name, r.role, r.relevance
LIMIT 20;
```

## 11. 实施建议

1. 先建约束和索引，再导入节点与关系。
2. 对于事实类数据，优先保证事实表完整，再计算汇总关系表。
3. 对于 `DIRECT_REL`、`COLLEAGUE_REL`、`ALUMNI_REL`、`INDIRECT_REL` 等结果关系，建议按批次重算。
4. 通过 `UpdateBatch` 和 `LOADED_IN_BATCH` 保留版本轨迹，满足审计和回溯要求。
5. 如果是正式生产环境，建议进一步补充字段字典、枚举值标准、规则版本说明和数据来源清单。

## 12. 总结

本知识图谱表体系的核心特点是：

1. 用基础实体节点表描述稳定对象。
2. 用事实节点表承载复杂上下文。
3. 用事实关系表保留可追溯的原始链路。
4. 用汇总/推理关系表支撑高效分析与可视化。

对于本项目而言，这种结构能够同时满足：

1. 专家关系构建
2. 合作成果分析
3. 同事/校友推理
4. 专家企业关联
5. 产业链事件研判
6. 全景图谱展示

如果后续需要进一步细化，可以继续补充两类文档：

1. 字段数据字典
