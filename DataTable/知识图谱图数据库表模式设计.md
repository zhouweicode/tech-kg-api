# 知识图谱图数据库表模式建议

## 1. 适用范围

本文基于《标书-知识图谱.docx》中的 9 个核心功能模块，整理适合亿级论文、专利、项目、学者、企业、产业链数据场景的图数据库表模式。  
表模式采用通用属性图模型，兼容 Neo4j、NebulaGraph、JanusGraph 一类图数据库。

## 2. 建模结论

建议采用“两层图谱模型”：

1. 事实层：保存论文作者、专利发明人、项目参与、工作经历、教育经历、企业合作、产业事件等原始事实。
2. 汇总/推理层：保存直接关系、同事关系、校友关系、论文合作关系、专家企业关系、间接关系、TOP-N 事件关联等派生结果。

这样设计的原因是：

1. 直接把所有信息压到“专家-专家”一条边上，会丢失时间、场景、成果编号、证据来源等上下文。
2. 工作经历、教育经历、合作事件本质上都带有大量属性，适合建成中间事实节点或富关系对象。
3. 上层分析查询需要快速返回结果，因此应保留汇总边，避免每次都从论文、专利、项目原始事实现场计算。

## 3. 总体建模原则

1. 主体实体做节点：学者、论文、专利、项目、机构、院校、企业、技术、产业链环节、事件等。
2. 关系上下文较多时，优先使用“事实节点 + 关系”而不是把所有属性都塞进一条边。
3. 原始事实与推理结果分开存储：
   原始事实保证可追溯，推理结果保证查询性能。
4. 所有核心实体都使用稳定业务主键：
   如 `scholar_id`、`paper_id`、`patent_id`、`project_id`、`org_id`、`enterprise_id`、`event_id`。
5. 所有推理边保留 `evidence_count`、`confidence`、`rule_version`、`version_no`、`updated_at` 等字段。

## 4. 基础节点 Tag 设计

| Tag 名称 | 主键/唯一键 | 关键属性 | 说明 |
| --- | --- | --- | --- |
| `Scholar` | `scholar_id` | `name`（姓名）、`alias_list`（别名列表）、`gender`（性别）、`title`（职称）、`research_fields`（研究方向）、`h_index`（H指数）、`citation_count`（总被引量）、`org_name_std`（标准机构名称） | 科技专家/人才主节点 |
| `Organization` | `org_id` | `org_name`（机构名称）、`org_type`（机构类型）、`region`（所在区域）、`uscc`（统一社会信用代码） | 高校、科研院所、政府机构等 |
| `Department` | `dept_id` | `dept_name`（部门名称）、`dept_type`（部门类型） | 机构下属院系、部门、团队 |
| `Paper` | `paper_id` | `title`（论文标题）、`doi`（DOI标识）、`publish_date`（发表日期）、`year`（发表年份）、`venue_id`（期刊会议ID）、`journal_level`（期刊会议级别）、`citation_count`（被引次数）、`keywords`（关键词） | 论文成果节点 |
| `Venue` | `venue_id` | `venue_name`（载体名称）、`venue_type`（载体类型）、`ccf_level`（CCF等级）、`sci_partition`（SCI分区）、`issn`（刊号） | 期刊/会议节点 |
| `Patent` | `patent_id` | `patent_no`（专利号）、`application_no`（申请号）、`title`（专利名称）、`apply_date`（申请日期）、`grant_date`（授权日期）、`ipc_cpc`（IPC/CPC分类）、`legal_status`（法律状态） | 专利成果节点 |
| `Project` | `project_id` | `project_no`（项目编号）、`project_name`（项目名称）、`project_type`（项目类型）、`start_date`（开始时间）、`end_date`（结束时间）、`funding`（项目经费）、`status`（项目状态） | 基金/科研项目节点 |
| `Enterprise` | `enterprise_id` | `enterprise_name`（企业名称）、`uscc`（统一社会信用代码）、`industry`（所属行业）、`region`（所在地区）、`establish_date`（成立时间）、`listed_status`（上市状态）、`specialized_tag`（企业标签）、`industry_rank`（行业地位） | 重点关注科技企业节点 |
| `School` | `school_id` | `school_name`（学校名称）、`region`（所在地区）、`school_type`（学校类型） | 院校主节点 |
| `College` | `college_id` | `college_name`（院系名称）、`discipline_group`（学科门类） | 院系/学院节点 |
| `Technology` | `tech_id` | `tech_name`（技术名称）、`tech_direction`（技术方向）、`tech_level`（技术层级）、`keywords`（技术关键词） | 技术领域或关键技术节点 |
| `IndustrySegment` | `segment_id` | `segment_name`（环节名称）、`segment_level`（环节层级）、`chain_position`（链上位置） | 产业链环节节点，上中下游可分层 |
| `Team` | `team_id` | `team_name`（团队名称）、`team_type`（团队类型）、`org_id`（所属机构ID）、`leader_id`（团队负责人ID） | 团队/课题组节点 |
| `IndustryEvent` | `event_id` | `event_name`（事件名称）、`event_type`（事件类型）、`event_time`（事件时间）、`segment_id`（所属环节ID）、`impact_score`（影响力评分）、`top_n_rank`（TOP-N排名）、`summary`（事件摘要） | 产业链关键事件节点 |
| `CapitalEvent` | `capital_event_id` | `investor_name`（投资方名称）、`amount`（融资金额）、`event_time`（事件时间）、`round`（融资轮次） | 融资/资本事件节点 |
| `Award` | `award_id` | `award_name`（奖项名称）、`award_level`（奖项等级）、`award_year`（获奖年份） | 奖项/评价信息节点 |
| `UpdateBatch` | `batch_id` | `version_no`（版本号）、`source_name`（数据来源名称）、`load_time`（导入时间）、`change_type`（变更类型）、`operator`（操作人） | 数据更新与回溯批次节点 |

## 5. 事实节点 Tag 设计

下列节点建议作为中间事实节点，用于保存复杂上下文。

| Tag 名称 | 主键/唯一键 | 关键属性 | 说明 |
| --- | --- | --- | --- |
| `Authorship` | `authorship_id` | `author_order`（作者顺序）、`is_corresponding`（是否通讯作者）、`org_name_raw`（原始署名单位）、`confidence`（识别置信度） | 学者参与论文的事实节点 |
| `Inventorship` | `inventorship_id` | `inventor_order`（发明人顺序）、`applicant_name_raw`（原始申请人名称）、`confidence`（识别置信度） | 学者参与专利的事实节点 |
| `ProjectParticipation` | `participation_id` | `role`（参与角色）、`responsibility`（职责说明）、`start_date`（参与开始时间）、`end_date`（参与结束时间）、`confidence`（识别置信度） | 学者参与项目的事实节点 |
| `Employment` | `employment_id` | `position`（岗位名称）、`title_level`（职级层级）、`start_date`（任职开始时间）、`end_date`（任职结束时间）、`status`（任职状态）、`source`（来源名称） | 学者任职经历节点 |
| `Education` | `education_id` | `major`（专业名称）、`degree`（学位）、`education_level`（培养层次）、`start_date`（入学时间）、`end_date`（毕业时间） | 学者教育经历节点 |
| `EnterpriseCooperation` | `coop_id` | `coop_type`（合作类型）、`role`（合作角色）、`tech_field`（技术领域）、`start_date`（合作开始时间）、`end_date`（合作结束时间）、`risk_tag`（风险标签）、`value_score`（价值评分） | 专家与企业合作事实节点 |
| `InferencePath` | `path_id` | `path_length`（路径长度）、`path_signature`（路径签名）、`strength_score`（关联强度评分）、`relation_label`（关系标签）、`rule_version`（规则版本）、`generated_at`（生成时间） | 间接关系推理路径节点 |
| `TrendReport` | `report_id` | `analysis_time`（分析时间）、`trend_direction`（趋势方向）、`risk_point`（风险点）、`opportunity_point`（机会点） | 事件影响分析结果节点 |

## 6. 基础边 Type 设计

### 6.1 事实边

| Edge Type | 起点 -> 终点 | 关键属性 | 说明 |
| --- | --- | --- | --- |
| `HAS_AUTHORSHIP` | `Scholar -> Authorship` | `source`（来源名称） | 学者拥有某篇论文作者事实 |
| `AUTHOR_OF` | `Authorship -> Paper` | `source`（来源名称） | 作者事实连接论文 |
| `PUBLISHED_IN` | `Paper -> Venue` | 无 | 论文发表于期刊/会议 |
| `HAS_INVENTORSHIP` | `Scholar -> Inventorship` | `source`（来源名称） | 学者拥有某项专利发明事实 |
| `INVENTOR_OF` | `Inventorship -> Patent` | `source`（来源名称） | 发明事实连接专利 |
| `HAS_PARTICIPATION` | `Scholar -> ProjectParticipation` | `source`（来源名称） | 学者拥有某个项目参与事实 |
| `PARTICIPATES_IN` | `ProjectParticipation -> Project` | `source`（来源名称） | 项目参与事实连接项目 |
| `HAS_EMPLOYMENT` | `Scholar -> Employment` | `source`（来源名称） | 学者任职事实 |
| `EMPLOYED_BY` | `Employment -> Organization` | `source`（来源名称） | 任职事实连接机构 |
| `IN_DEPARTMENT` | `Employment -> Department` | `source`（来源名称） | 任职事实连接部门/团队 |
| `HAS_EDUCATION` | `Scholar -> Education` | `source`（来源名称） | 学者教育事实 |
| `STUDIED_AT` | `Education -> School` | `source`（来源名称） | 教育事实连接学校 |
| `IN_COLLEGE` | `Education -> College` | `source`（来源名称） | 教育事实连接院系 |
| `HAS_ENTERPRISE_COOP` | `Scholar -> EnterpriseCooperation` | `source`（来源名称） | 学者与企业合作事实 |
| `COOP_WITH_ENTERPRISE` | `EnterpriseCooperation -> Enterprise` | `source`（来源名称） | 企业合作事实连接企业 |
| `RELATED_TECH` | `EnterpriseCooperation -> Technology` | `weight`（权重） | 合作涉及技术领域 |
| `OWNS_TECH` | `Enterprise -> Technology` | `importance`（重要度） | 企业掌握/布局技术 |
| `FOCUSES_ON` | `Scholar -> Technology` | `weight`（权重） | 学者研究方向 |
| `BELONGS_TO_SEGMENT` | `Enterprise -> IndustrySegment` | `position`（环节位置） | 企业所属产业链环节 |
| `PART_OF_SEGMENT` | `Technology -> IndustrySegment` | `position`（环节位置） | 技术所属产业链环节 |
| `EVENT_IN_SEGMENT` | `IndustryEvent -> IndustrySegment` | `impact_score`（影响力评分） | 事件发生于某产业环节 |
| `EVENT_INVOLVES_TECH` | `IndustryEvent -> Technology` | `relevance`（相关度） | 事件关联技术 |
| `INVESTS_IN` | `CapitalEvent -> Enterprise` | `amount`（融资金额）、`event_time`（事件时间） | 资本事件关联企业 |
| `CREATED_IN_BATCH` | `任意节点/边 -> UpdateBatch` | `op_type`（操作类型） | 更新批次与版本追踪 |

### 6.2 汇总/推理边

| Edge Type | 起点 -> 终点 | 关键属性 | 说明 |
| --- | --- | --- | --- |
| `DIRECT_REL` | `Scholar -> Scholar` | `relation_types`（关系类型列表）、`coop_count`（合作次数）、`first_time`（首次合作时间）、`last_time`（最近合作时间）、`evidence_count`（证据数量）、`confidence`（关系置信度） | 直接关系汇总边 |
| `PAPER_COOP_REL` | `Scholar -> Scholar` | `paper_count`（合作论文数）、`first_year`（首次合作年份）、`last_year`（最近合作年份）、`citation_sum`（累计被引量）、`team_flag`（是否核心团队）、`influence_score`（影响力评分） | 论文合作汇总边 |
| `COLLEAGUE_REL` | `Scholar -> Scholar` | `org_id`（机构ID）、`dept_id`（部门ID）、`overlap_start`（重叠开始时间）、`overlap_end`（重叠结束时间）、`overlap_days`（重叠天数）、`evidence_count`（证据数量） | 同事关系汇总边 |
| `ALUMNI_REL` | `Scholar -> Scholar` | `school_id`（学校ID）、`college_id`（院系ID）、`relation_subtype`（关系子类型）、`entry_year_gap`（入学年份差）、`evidence_count`（证据数量） | 校友/院友关系汇总边 |
| `EXPERT_ENTERPRISE_REL` | `Scholar -> Enterprise` | `relation_type`（关系类型）、`role`（角色）、`tech_field`（技术领域）、`start_date`（开始时间）、`end_date`（结束时间）、`risk_tag`（风险标签）、`value_score`（价值评分） | 专家-企业汇总边 |
| `INDIRECT_REL` | `Scholar -> Scholar` | `min_path_length`（最短路径长度）、`strength_score`（关联强度评分）、`relation_label`（关系标签）、`path_count`（路径数量）、`rule_version`（规则版本）、`generated_at`（生成时间） | 间接关系汇总边 |
| `PAIR_COOP_SUMMARY` | `Scholar -> Scholar` | `paper_count`（合作论文数）、`patent_count`（合作专利数）、`project_count`（合作项目数）、`award_count`（获奖数量）、`coop_mode`（合作模式）、`coop_depth_score`（合作深度评分） | 两点合作成果汇总边 |
| `EVENT_EXPERT_REL` | `IndustryEvent -> Scholar` | `role`（角色）、`relevance`（相关度）、`evidence_count`（证据数量） | 事件关联专家 |
| `EVENT_TEAM_REL` | `IndustryEvent -> Team` | `role`（角色）、`relevance`（相关度） | 事件关联团队 |
| `UPSTREAM_OF` | `IndustrySegment -> IndustrySegment` | `weight`（权重） | 产业链上下游关系 |
| `VALUE_FLOW_TO` | `IndustrySegment -> IndustrySegment` | `flow_value`（流向值）、`flow_type`（流向类型）、`update_time`（更新时间） | 产业链价值流向 |

## 7. 主要功能对应的图数据库表模式

### 7.1 科技专家/人才直接关系

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `Scholar`（专家）、`Paper`（论文）、`Patent`（专利）、`Project`（项目）、`Authorship`（作者事实）、`Inventorship`（发明事实）、`ProjectParticipation`（项目参与事实） |
| 核心 Edge | `HAS_AUTHORSHIP`（拥有作者事实）、`AUTHOR_OF`（作者对应论文）、`HAS_INVENTORSHIP`（拥有发明事实）、`INVENTOR_OF`（发明对应专利）、`HAS_PARTICIPATION`（拥有项目参与事实）、`PARTICIPATES_IN`（参与项目）、`DIRECT_REL`（直接关系） |
| 关键字段 | `relation_types`（关系类型列表）、`coop_count`（合作次数）、`first_time`（首次合作时间）、`last_time`（最近合作时间）、`evidence_count`（证据数量）、`evidence_ids`（成果标识符列表）、`confidence`（关系置信度） |
| 说明 | 原始事实来自论文/专利/项目参与，中间计算后物化为 `DIRECT_REL`，便于快速查询与溯源 |

### 7.2 科技单节点间接关系

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `Scholar`（专家）、`InferencePath`（推理路径） |
| 核心 Edge | `DIRECT_REL`（直接关系）、`COLLEAGUE_REL`（同事关系）、`ALUMNI_REL`（校友关系）、`PAPER_COOP_REL`（论文合作关系）、`EXPERT_ENTERPRISE_REL`（专家企业关系）、`INDIRECT_REL`（间接关系） |
| 关键字段 | `min_path_length`（最短路径长度）、`strength_score`（关联强度评分）、`path_count`（路径数量）、`relation_label`（关系标签）、`path_signature`（路径签名）、`rule_version`（规则版本） |
| 说明 | `INDIRECT_REL` 保存摘要结果，`InferencePath` 保存可视化展示所需完整路径与推理依据 |

### 7.3 科技两点合作成果

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `Scholar`（专家）、`Paper`（论文）、`Patent`（专利）、`Project`（项目）、`Award`（奖项） |
| 核心 Edge | `DIRECT_REL`（直接关系）、`PAIR_COOP_SUMMARY`（两点合作成果汇总）、`HAS_AUTHORSHIP`（拥有作者事实）、`AUTHOR_OF`（作者对应论文）、`HAS_INVENTORSHIP`（拥有发明事实）、`INVENTOR_OF`（发明对应专利）、`HAS_PARTICIPATION`（拥有项目参与事实）、`PARTICIPATES_IN`（参与项目） |
| 关键字段 | `paper_count`（合作论文数）、`patent_count`（合作专利数）、`project_count`（合作项目数）、`award_count`（获奖数量）、`coop_mode`（合作模式）、`coop_depth_score`（合作深度评分）、`year_dist`（年份分布）、`tech_dist`（技术分布） |
| 说明 | 明细成果仍由事实层返回，`PAIR_COOP_SUMMARY` 负责加速统计分析与报告生成 |

### 7.4 科技专家同事关系

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `Scholar`（专家）、`Employment`（任职经历）、`Organization`（机构）、`Department`（部门） |
| 核心 Edge | `HAS_EMPLOYMENT`（拥有任职事实）、`EMPLOYED_BY`（任职于机构）、`IN_DEPARTMENT`（任职于部门）、`COLLEAGUE_REL`（同事关系） |
| 关键字段 | `org_id`（机构ID）、`dept_id`（部门ID）、`overlap_start`（重叠开始时间）、`overlap_end`（重叠结束时间）、`overlap_days`（重叠天数）、`team_background`（团队背景）、`evidence_count`（证据数量） |
| 说明 | 同事关系由任职时间重叠规则计算得到，原始任职事实必须保留以支持回溯校验 |

### 7.5 科技专家校友关系

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `Scholar`（专家）、`Education`（教育经历）、`School`（学校）、`College`（院系） |
| 核心 Edge | `HAS_EDUCATION`（拥有教育事实）、`STUDIED_AT`（就读于学校）、`IN_COLLEGE`（就读于院系）、`ALUMNI_REL`（校友关系） |
| 关键字段 | `school_id`（学校ID）、`college_id`（院系ID）、`relation_subtype`（关系子类型）、`entry_year_gap`（入学年份差）、`education_level`（培养层次）、`evidence_count`（证据数量） |
| 说明 | 校友、院友建议统一为 `ALUMNI_REL`，通过 `relation_subtype` 细分，避免边类型膨胀 |

### 7.6 科技专家论文合作关系

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `Scholar`（专家）、`Paper`（论文）、`Authorship`（作者事实）、`Venue`（期刊会议）、`Technology`（技术） |
| 核心 Edge | `HAS_AUTHORSHIP`（拥有作者事实）、`AUTHOR_OF`（作者对应论文）、`PUBLISHED_IN`（发表于载体）、`FOCUSES_ON`（研究方向聚焦）、`PAPER_COOP_REL`（论文合作关系） |
| 关键字段 | `paper_count`（合作论文数）、`citation_sum`（累计被引量）、`journal_level_dist`（期刊级别分布）、`common_topics`（共同主题）、`team_flag`（是否核心团队）、`influence_score`（影响力评分） |

### 7.7 重点关注科技企业关系

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `Scholar`（专家）、`Enterprise`（企业）、`EnterpriseCooperation`（企业合作事实）、`Patent`（专利）、`Project`（项目）、`Technology`（技术） |
| 核心 Edge | `HAS_ENTERPRISE_COOP`（拥有企业合作事实）、`COOP_WITH_ENTERPRISE`（与企业合作）、`RELATED_TECH`（关联技术）、`EXPERT_ENTERPRISE_REL`（专家企业关系）、`OWNS_TECH`（掌握技术） |
| 关键字段 | `relation_type`（关系类型）、`role`（角色）、`tech_field`（技术领域）、`start_date`（开始时间）、`end_date`（结束时间）、`risk_tag`（风险标签）、`value_score`（价值评分） |
| 说明 | 企业关系既有任职/顾问，也有研发合作，建议保留事实节点 `EnterpriseCooperation` 承载复杂上下文 |

### 7.8 科技产业链点 TOP-N 事件关系

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `IndustryEvent`（产业事件）、`IndustrySegment`（产业链环节）、`Technology`（技术）、`Scholar`（专家）、`Team`（团队）、`TrendReport`（趋势报告） |
| 核心 Edge | `EVENT_IN_SEGMENT`（事件发生于环节）、`EVENT_INVOLVES_TECH`（事件关联技术）、`EVENT_EXPERT_REL`（事件关联专家）、`EVENT_TEAM_REL`（事件关联团队） |
| 关键字段 | `impact_score`（影响力评分）、`top_n_rank`（TOP-N排名）、`role`（角色）、`relevance`（相关度）、`trend_direction`（趋势方向）、`risk_point`（风险点）、`opportunity_point`（机会点） |
| 说明 | 事件节点保存排名与评分，趋势判断可独立为 `TrendReport` 节点，避免把长文本结论写进边属性 |

### 7.9 科技产业链全景图

| 类别 | 表模式 |
| --- | --- |
| 核心 Tag | `IndustrySegment`（产业链环节）、`Enterprise`（企业）、`Technology`（技术）、`Scholar`（专家）、`CapitalEvent`（资本事件） |
| 核心 Edge | `BELONGS_TO_SEGMENT`（属于产业链环节）、`PART_OF_SEGMENT`（技术属于环节）、`OWNS_TECH`（掌握技术）、`FOCUSES_ON`（研究方向聚焦）、`UPSTREAM_OF`（上下游关系）、`VALUE_FLOW_TO`（价值流向）、`INVESTS_IN`（投资于企业） |
| 关键字段 | `chain_position`（链上位置）、`weight`（权重）、`flow_value`（流向值）、`flow_type`（流向类型）、`amount`（金额）、`update_time`（更新时间） |
| 说明 | 产业链全景图以环节为骨架，以企业、技术、人才、资本为关键要素，重点支持钻取、筛选、价值流向展示 |

## 8. 索引、约束与分层存储建议

### 8.1 必建唯一索引/约束

| 对象 | 建议字段 |
| --- | --- |
| `Scholar` | `scholar_id` |
| `Paper` | `paper_id`、`doi` |
| `Patent` | `patent_id`、`patent_no`、`application_no` |
| `Project` | `project_id`、`project_no` |
| `Organization` | `org_id` |
| `Enterprise` | `enterprise_id`、`uscc` |
| `School` | `school_id` |
| `Technology` | `tech_id` |
| `IndustrySegment` | `segment_id` |
| `IndustryEvent` | `event_id` |
| `UpdateBatch` | `batch_id`、`version_no` |

### 8.2 建议检索索引

| 对象 | 建议字段 | 用途 |
| --- | --- | --- |
| `Scholar` | `name`、`org_name_std` | 专家检索、同名消歧辅助 |
| `Paper` | `year`、`title` | 成果筛选、时间过滤 |
| `Patent` | `grant_date`、`ipc_cpc` | 专利分析、技术领域过滤 |
| `Project` | `start_date`、`project_type` | 项目范围筛选 |
| `Enterprise` | `industry`、`region` | 企业专题分析 |
| `IndustryEvent` | `event_time`、`top_n_rank` | 事件排序与查询 |
| 汇总边 | `updated_at`、`confidence`、`strength_score` | 推理结果过滤 |

### 8.3 存储建议

1. 原始全文、长摘要、报告正文不建议全部放进图数据库，可存对象存储或检索引擎，只在图中保留主键和摘要。
2. 批量统计、画像宽表、报表快照建议沉到关系库或列式库，图数据库聚焦关系检索和路径分析。
3. `DIRECT_REL`、`PAPER_COOP_REL`、`COLLEAGUE_REL`、`ALUMNI_REL` 等汇总边建议按日或按批次重算。
4. `UpdateBatch` 必须与推理结果关联，满足“动态更新、日志留痕、版本回溯”的标书要求。

## 9. 推荐落地方式

### 9.1 如果采用 Neo4j

1. 适合做路径查询、可视化联动、推理结果快速验证。
2. 适合将 `InferencePath`、`Employment`、`Education`、`EnterpriseCooperation` 设计为中间节点。

### 9.2 如果采用 NebulaGraph

1. 建议按 `Tag + Edge Type` 方式落地上述表模式。
2. 可将时间序列型汇总边使用 `rank` 区分不同版本或不同年度快照。
3. 索引只建在高频入口字段上，避免大规模写入时索引过重。
